3. Luồng Login

User mở app lần đầu, chưa có session hợp lệ, đi qua Login page rồi vào HubPro.

1User mở localhost:3000
Browser load React app, AuthGuard mount trước mọi route.
↓
2AuthGuard check session
authClient.getSession() đọc localStorage key sb-fghrcp...-auth-token. Chưa login → null.
↓
3Redirect /login?next=...
Login page hiện form email + password. URL giữ ?next= để redirect về sau login.
↓
4Submit → signInWithPassword()
Gửi POST /auth/v1/token tới Project A với email + password.
↓
5Project A trả JWT
Response gồm access_token (JWT) + refresh_token. Client lưu vào localStorage.
↓
6AuthGuard fetch profile
SELECT role, allowed_tools FROM profiles WHERE id = user_id. Role authenticated, RLS pass vì user đọc profile của chính mình.
↓
7Render HubPro
Admin → full tool list. User → chỉ tool trong allowed_tools.

4. Luồng Create User

Admin dùng Setting → User Management → nút "Create user". Frontend gọi Edge Function, function chạy dưới role service_role.

1Admin điền form
Email, password, tick tool cho user mới → Save.
↓
2POST /functions/v1/create-user
Header Authorization: Bearer <admin JWT>. Body JSON email + password + tools.
↓
3verifyAdmin trong function
Decode JWT → user_id. Query SELECT role FROM profiles dưới role service_role. Nếu role !== 'admin' → 403.
↓
4adminClient.auth.admin.createUser()
Tạo row auth.users với password hash. Trả về UUID mới.
↓
5INSERT INTO profiles
id = UUID mới, role, allowed_tools. Nếu fail → rollback bằng cách xoá row auth.users.
↓
6200 OK
Frontend refresh user list, user mới xuất hiện.

Verify admin — 2 tầng permission

TầngCâu hỏiCơ chếFix khi fail
Tầng 1: Privilege Role service_role có được đụng bảng profiles không? GRANT SELECT ON TABLE profiles TO service_role Chạy GRANT trong SQL Editor
Tầng 2: RLS Role được đọc row nào trong bảng? CREATE POLICY ON profiles service_role bypass RLS, không cần policy

5. Role trong PostgreSQL / Supabase

RoleAi dùngKeyQuyền
anon Visitor chưa login anon key (public, ok expose FE) Đọc public data, giới hạn
authenticated User login qua Supabase Auth JWT access_token Đọc/ghi theo RLS policy
service_role Backend / Edge Function service_role key (SECRET) Bypass RLS, cần GRANT tay
supabase_admin / postgres Supabase internal Không expose ra ngoài Superuser, DDL toàn quyền
Rule cứng: service_role key KHÔNG bao giờ paste vào frontend, không commit git. Chỉ đặt trong Edge Function Secrets ở Supabase Dashboard. Leak = attacker toàn quyền DB.

6. Storage của session

Sau login, Supabase client lưu session trong localStorage của browser. Mỗi project 1 key riêng, không đè nhau.

Project A

sb-fghrcp...-auth-token

Chứa: JWT access_token, refresh_token, user object. Auto-refresh token khi expire.

Project B

sb-vlerovuj...-auth-token

Session riêng cho Library/RAG nếu tool đó cần login vault. Có thể null nếu Library dùng public bucket.

Debug tip: nếu app confuse session, F12 → Application → Local Storage → xoá hết key có prefix sb-, sau đó reload + login lại. Session tự tạo mới clean.
